## 1. 系统概述

运维辅助系统（Ops Assistant）是一款面向运维团队的智能故障处理平台，核心理念是：

| 能力 | 说明 |
|------|------|
| **快速定位** | 输入故障描述，AI 自动识别问题模块、版本和部署模式，推荐历史相似案例 |
| **工单流转** | 标准化工单生命周期管理：待处理 → 处理中 → 已解决 → 已关闭 |
| **知识沉淀** | 工单解决后可一键生成知识库条目，积累团队排查经验 |
| **排查指引** | 按模块和部署模式维护标准化排查路径，新人也能快速上手 |

### 技术架构

```
┌─────────────┐     ┌─────────────┐     ┌──────────┐
│   React 前端  │────▶│ FastAPI 后端  │────▶│  MySQL   │
│  (Ant Design) │     │  (Python)    │     │   8.0    │
└─────────────┘     └──────┬───────┘     └──────────┘
                           │
                    ┌──────▼───────┐
                    │  LLM API     │
                    │ (可配置模型)   │
                    └──────────────┘
```

## 2. 部署与启动

### 2.1 环境要求

| 组件 | 最低版本 |
|------|----------|
| Python | 3.12+ |
| Node.js | 18+ |
| MySQL | 8.0+ |

### 2.2 Docker 一键启动（推荐）

```bash
# 启动所有服务（MySQL + 后端 + 前端）
docker-compose up -d

# 首次启动后初始化数据库
docker-compose exec backend python init_db.py
```

### 2.3 手动启动

#### 后端

```bash
cd backend

# 创建虚拟环境
python -m venv .venv
source .venv/bin/activate     # Linux/macOS
.venv\Scripts\activate        # Windows

# 安装依赖
pip install -r requirements.txt

# 配置 .env 文件（参考 .env.example）
# 必填项：SECRET_KEY, DATABASE_URL
# AI 功能需要：LLM_API_KEY, LLM_API_BASE, LLM_MODEL

# 初始化数据库
python init_db.py

# 启动服务
uvicorn app.main:app --host 0.0.0.0 --port 8000 --reload
```

#### 前端

```bash
cd frontend
npm install
npm run dev
```

### 2.4 访问地址

| 服务 | 地址 |
|------|------|
| 前端页面 | http://localhost:3001 |
| 后端 API | http://localhost:8000 |
| API 交互文档 | http://localhost:8000/docs |

### 2.5 环境变量说明

编辑 `backend/.env`：

```ini
# ---- 必填 ----
SECRET_KEY=<强随机字符串>              # JWT 签名密钥，生产环境务必修改
DATABASE_URL=mysql+aiomysql://<user>:<pass>@<host>:<port>/engineer_support

# ---- AI 功能（不填则使用关键词匹配降级模式）----
LLM_API_KEY=<你的 API Key>
LLM_API_BASE=https://api.openai.com/v1    # OpenAI 兼容的 API 地址
LLM_MODEL=gpt-4o-mini                      # 模型名称

# ---- 可选 ----
ACCESS_TOKEN_EXPIRE_MINUTES=1440           # Token 有效期（分钟），默认 24h
MAX_UPLOAD_SIZE_MB=50                      # 附件上传大小上限
CORS_ORIGINS=*                             # 允许的跨域来源，* 表示全部
```

> **密码含特殊字符注意**：`DATABASE_URL` 中的 `!`、`@`、`:` 等字符需 URL 编码（`!` → `%21`，`@` → `%40`）。

## 3. 用户角色与权限

| 角色 | 权限范围 |
|------|---------|
| **管理员 (admin)** | 所有功能：工单管理、智能定位、新建/审核知识条目、管理后台（客户/部署/模块/版本/排查路径/用户） |
| **运维人员 (operator)** | 工单管理、智能定位、查看知识库、新建知识条目（初始状态为"草稿"，需管理员审核） |

**默认管理员账号**：用户名 `admin`，密码 `admin123`（首次登录后务必修改）。

### 4.3 工单管理

#### 4.3.1 工单列表

> 访问地址：`/tickets`

**筛选条件**：

| 筛选项 | 可选值 | 说明 |
|--------|--------|------|
| 搜索框 | 自由文本 | 搜索标题包含关键词的工单 |
| 状态 | 待处理 / 处理中 / 已解决 / 已关闭 | |
| 优先级 | p0 / p1 / p2 / p3 | |

**列说明**：

| 列名 | 说明 |
|------|------|
| 优先级 | `p0`（红色，最高）/ `p1`（橙色）/ `p2`（蓝色，默认）/ `p3`（绿色） |
| 标题 | 工单标题 |
| 状态 | 当前处理状态 |
| 模块 | 关联的问题模块 |
| 创建时间 | 工单创建时间 |

**操作**：
- 点击任意行 → 进入工单详情页
- 点击「新建工单」→ 创建新工单
- 列表每 30 秒自动刷新

#### 4.3.2 创建工单

> 访问地址：`/tickets/new`

**表单字段**：

| 字段 | 是否必填 | 可选值 | 说明 |
|------|---------|--------|------|
| 标题 | ✅ | 自由文本 | 简洁描述问题 |
| 描述 | ✅ | 多行文本 | 详细描述故障现象 |
| 客户 | 否 | 下拉选择 | 关联已有客户 |
| 部署 | 否 | 下拉选择 | 关联部署实例 |
| 模块 | 否 | 下拉选择 | 关联故障模块 |
| 版本 | 否 | 下拉选择 | 关联版本号 |
| 部署模式 | 否 | 单机 / HA / 集群 / 上下级 | 客户部署架构 |
| 优先级 | ✅ | P0 / P1 / P2 / P3 | 严重程度 |
| 来源 | 否 | Jira / 企微 / 手动 | 工单来源渠道 |

> 提交后自动跳转到工单列表页。

#### 4.3.3 工单详情与处理

> 访问地址：`/tickets/:id`

页面分为左右两栏：

**左栏 — 工单信息**：

展示工单的完整属性（描述、客户、模块、版本、部署模式、优先级、状态、创建时间、来源）。

**右栏 — 处理操作**：

| 区域 | 功能 | 说明 |
|------|------|------|
| **故障排查清单** | 勾选检查项 | 标准化排查步骤，按顺序逐项确认 |
| **解决方案** | 文本录入 | 记录解决方案，可与状态流转配合使用 |
| **状态操作** | 状态流转按钮 | 见下方状态机说明 |
| **附件** | 待接入 | 上传日志、截图、配置文件等 |
| **生成案例** | 一键沉淀 | 基于当前工单用 AI 生成知识库草案 |

**工单状态流转规则**：

```
pending ──→ processing ──→ resolved ──→ closed
  ↑             ↑              │            │
  └─── 不可回退 ──┘             └──── 可终止 ────┘

完整路径：
  pending → processing → resolved → closed
  pending → processing → closed（直接关闭，跳过解决）
```

具体操作按钮：

| 按钮 | 可用条件 | 效果 |
|------|---------|------|
| **处理中** | 当前状态 ≠ processing | 认领工单，状态变为 processing |
| **已解决** | 当前状态 ≠ resolved | 标记已解决，记录解决方案，状态变为 resolved |
| **已关闭** | 当前状态 ≠ closed | 关闭工单，状态变为 closed |

#### 4.3.4 生成案例（知识沉淀）

在工单详情页点击「生成案例」，系统会：

1. 调用 LLM 将工单的**描述 + 根因 + 解决方案**转化为结构化知识条目
2. 自动生成案例标题、根因分析、解决方案和标签
3. 将案例以"草稿"状态存入知识库

生成后可在知识库中查看和编辑。

### 4.4 智能定位

> 访问地址：`/ai-locate`

这是系统的核心 AI 功能，帮助运维人员快速缩小排查范围。

#### 使用步骤

1. **输入问题描述**：在文本框中详细描述故障现象（越详细越准确）
2. **点击「开始定位」**：系统调用 LLM 分析问题
3. **查看分析结果**：

#### 定位结果包含四个区域

**① 识别结果**

| 识别项 | 说明 |
|--------|------|
| 识别模块 | AI 判断最可能出问题的模块名称 |
| 识别版本 | AI 推测涉及的版本号 |
| 部署模式 | AI 识别的部署架构类型 |
| 根因候选 | AI 推测的可能根因列表（按置信度排序） |

**② 推荐方案**

- 匹配知识库中相似的历史案例
- 每个案例显示标题、部署模式、置信度百分比
- 点击案例可跳转知识库详情

**③ 排查路径**

- 展示该模块和部署模式对应的标准化排查步骤
- 可逐项勾选确认已完成的操作
- 如果没有匹配的排查路径，显示"暂无排查路径"

> **降级模式**：如果未配置 LLM API Key，系统会退化为关键词匹配模式（根据描述中的模块名和版本号做简单匹配）。

### 4.5 知识库

知识库是运维经验的沉淀中心，存储从工单生成的案例和手工录入的排查知识。

#### 4.5.1 知识列表

> 访问地址：`/knowledge`

**筛选条件**：

| 筛选项 | 可选值 | 说明 |
|--------|--------|------|
| 搜索框 | 自由文本 | 模糊搜索标题 |
| 模块 | 已录入的模块列表 | 按模块过滤 |
| 部署模式 | 单机 / HA / 集群 / 上下级 | 按部署架构过滤 |
| 审核状态 | 草稿 / 已审核 / 已归档 | 按审核状态过滤 |

**列说明**：

| 列名 | 说明 |
|------|------|
| 标题 | 知识条目标题 |
| 模块 | 关联模块 |
| 部署模式 | 关联部署架构 |
| 标签 | 关键词标签 |
| 审核状态 | 草稿（橙色）/ 已审核（绿色）/ 已归档（蓝色） |
| 创建时间 | 创建日期时间 |

#### 4.5.2 新建知识条目

> 访问地址：`/knowledge/new`

> ⚠️ **前置条件**：知识条目必须关联一个已存在的模块。如果下拉框中没有模块可选，请先去「管理后台 → 模块管理」创建模块，再回来新建知识条目。

**表单字段**：

| 字段 | 是否必填 | 说明 |
|------|---------|------|
| 标题 | ✅ | 知识条目标题，如"XX模块在HA模式下日志丢失问题" |
| 模块 | ✅ | 输入模块标识 |
| 部署模式 | 否 | 单机 / HA / 集群 / 上下级 |
| 根因分析 | 否 | 详细根因描述 |
| 解决方案 | 否 | 分步骤的解决方案 |
| 标签 | 否 | 逗号分隔，如"网络,数据库,配置" |
| 关联工单 | 否 | 输入工单 UUID |
| 排查路径 | 否 | 动态添加排查步骤（每步含标题+描述） |

#### 4.5.3 知识详情与审核

> 访问地址：`/knowledge/:id`

**展示内容**：
- 标题 + 审核状态标签 + 关键词标签
- 模块、部署模式、创建/更新时间、关联工单、置信度
- 根因分析、解决方案、排查路径

**管理员审核操作**（仅 admin 可见）：

| 操作 | 说明 |
|------|------|
| **审核通过** | 案例状态变为 reviewed，可被推荐系统使用 |
| **归档** | 案例状态变为 archived，保留但暂不推荐 |

案例生命周期：`draft → reviewed → archived`

## 5. 完整操作流程指南

### 场景一：收到客户故障 → 定位 → 解决 → 沉淀

```
Step 1  ──  登录系统
Step 2  ──  工作台 → 📝 新建工单
            │  填写标题、描述、优先级、部署模式等
            │  提交
Step 3  ──  进入工单详情页
            │  查看排查清单，逐项确认
Step 4  ──  使用 🔍 智能定位
            │  粘贴故障描述 → 开始定位
            │  查看 AI 识别的模块、根因、相似案例
            │  参考推荐方案和排查路径
Step 5  ──  回到工单详情页
            │  填写解决方案
            │  点击「已解决」
Step 6  ──  点击「生成案例」
            │  AI 自动转为知识库条目（草稿）
Step 7  ──  管理员审核知识条目
            │  知识库 → 找到草稿条目
            │  审核通过 / 归档
```

### 场景二：直接查阅知识库解决重复问题

```
Step 1  ──  登录系统
Step 2  ──  📚 知识库
            │  按模块 / 部署模式筛选
            │  或搜索关键词
Step 3  ──  找到匹配案例
            │  查看根因分析、解决方案、排查路径
            │  直接按步骤操作
```

### 场景三：手工录入排查经验（无工单场景）

```
Step 1  ──  知识库 → 新建
Step 2  ──  填写标题、模块、部署模式
Step 3  ──  填写根因分析、解决方案
Step 4  ──  添加标签和排查路径步骤
Step 5  ──  提交（状态 = draft）
Step 6  ──  管理员审核通过
```

## 6. 数据字典

### 6.1 工单（Ticket）

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | UUID | 唯一标识 |
| `external_id` | string | 外部系统 ID（如 Jira 工单号） |
| `title` | string(200) | 工单标题 |
| `description` | text | 问题描述 |
| `status` | enum | `pending` / `processing` / `resolved` / `closed` |
| `priority` | enum | `p0`（紧急）/ `p1`（高）/ `p2`（中）/ `p3`（低） |
| `source` | enum | `jira` / `wechat` / `manual` |
| `module_id` | FK | 关联模块 |
| `version_id` | FK | 关联版本 |
| `deploy_mode` | enum | `standalone` / `ha` / `cluster` / `hierarchical` |
| `customer_id` | FK | 关联客户 |
| `deployment_id` | FK | 关联部署实例 |
| `assignee_id` | FK | 指派处理人 |
| `identified_root_cause` | text | 识别的根因 |
| `solution` | text | 解决方案 |
| `troubleshooting_checklist` | JSON | 排查清单 |
| `auto_identified` | bool | 是否 AI 自动识别 |
| `created_at` | datetime | 创建时间 |
| `resolved_at` | datetime | 解决时间 |
| `updated_at` | datetime | 更新时间 |

### 6.2 知识案例（Case）

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | UUID | 唯一标识 |
| `ticket_id` | FK (nullable) | 关联工单 |
| `title` | string(200) | 案例标题 |
| `module_id` | FK | 关联模块 |
| `deploy_mode` | enum | 部署模式 |
| `root_cause` | text | 根因分析 |
| `solution` | text | 解决方案 |
| `troubleshooting_path` | JSON | 排查路径步骤列表 |
| `tags` | JSON | 标签列表 |
| `confidence_score` | float | 置信度评分 (0.0–1.0) |
| `review_status` | enum | `draft` / `reviewed` / `archived` |
| `reviewed_by` | FK | 审核人 |
| `created_by` | FK | 创建人 |
| `customer_id` | FK (nullable) | 关联客户 |

### 6.3 排查路径（TroubleshootingPath）

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | UUID | 唯一标识 |
| `module_id` | FK | 关联模块 |
| `deploy_mode` | enum | `all` / `standalone` / `ha` / `cluster` / `hierarchical` |
| `steps` | JSON | 排查步骤列表 `[{title, description}]` |
| `version` | string | 路径版本号 |

> `deploy_mode = "all"` 表示该排查路径适用于所有部署模式。AI 定位时会先匹配特定模式，未找到时 fallback 到 "all"。

### 6.4 枚举值汇总

| 枚举名称 | 可选值 |
|---------|--------|
| **工单状态** | `pending`（待处理）、`processing`（处理中）、`resolved`（已解决）、`closed`（已关闭） |
| **优先级** | `p0`（紧急）、`p1`（高）、`p2`（中）、`p3`（低） |
| **来源** | `jira`、`wechat`（企微）、`manual`（手动） |
| **部署模式** | `standalone`（单机）、`ha`（HA）、`cluster`（集群）、`hierarchical`（上下级） |
| **环境** | `test`（测试）、`staging`（预发布）、`production`（生产） |
| **合同级别** | `basic`（基础）、`standard`（标准）、`vip`（VIP） |
| **审核状态** | `draft`（草稿）、`reviewed`（已审核）、`archived`（已归档） |
| **用户角色** | `admin`（管理员）、`operator`（运维人员） |

## 8. 常见问题

| 问题 | 原因 | 解决 |
|------|------|------|
| 登录后闪回登录页 | 数据库未初始化 | 运行 `python init_db.py` |
| 500 Internal Server Error | 数据库连接失败 | 检查 `backend/.env` 中 `DATABASE_URL` |
| 智能定位返回结果不准确 | 未配置 LLM 或 prompt 描述不详细 | 确认 LLM_API_KEY 已配置，使用更详细的描述 |
| 智能定位提示"暂无匹配案例" | 知识库中缺少相关案例 | 先在知识库中录入相关案例 |
| 工单状态流转失败（400 错误） | 状态流转不合法 | 按 pending→processing→resolved→closed 顺序操作 |
| 创建案例失败 | 模块未创建或必填字段缺失 | 确保已先在管理后台中创建了模块，再填写标题和模块 |
| AI 功能不可用 | LLM API Key 未配置 | 配置 `.env` 中的 LLM_API_KEY 和 LLM_API_BASE |
| 配置 LLM 后仍不工作 | API 地址或 Key 格式错误 | 确认 LLM_API_BASE 格式为 `https://xxx.com/v1` |
| 附件上传不可用 | 功能待接入 | 后续版本将完善附件功能 |
| docker-compose 启动后端口冲突 | 3306/8000/3000 被占用 | 修改 docker-compose.yml 中的端口映射 |
